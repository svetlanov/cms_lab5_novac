## Контрольные вопросы

### 1. Почему DISALLOW_FILE_EDIT и правильные права на wp-config.php существенно уменьшают риск пост-эксплойта?

`DISALLOW_FILE_EDIT` запрещает редактирование файлов тем и плагинов через интерфейс WordPress. Если злоумышленник получит доступ к аккаунту администратора (например, через подбор пароля), он не сможет внедрить вредоносный код в файлы сайта через браузер — для этого ему потребуется прямой доступ к серверу по FTP или SSH.

Правильные права на `wp-config.php` (640 или 600) означают, что файл недоступен для чтения другими пользователями системы. В `wp-config.php` хранятся логин и пароль от базы данных, секретные ключи и соли. Если злоумышленник получит доступ к этому файлу, он получит полный контроль над базой данных сайта.

Вместе эти две меры закрывают два ключевых вектора атаки после первоначального взлома (пост-эксплойт): модификацию кода через браузер и кражу учётных данных БД.

### 2. Какие параметры Login Lockdown/Firewall выбраны и почему?

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Max Login Attempts | 5 | Достаточно для случайных опечаток, но слишком мало для автоматического перебора |
| Login Retry Time Period | 15 мин | Сброс счётчика через 15 минут — баланс между безопасностью и удобством |
| Lockout Time | 30 мин | Достаточно долго чтобы остановить атаку, но не настолько долго чтобы заблокировать легитимного пользователя надолго |
| Force Logout | 24 ч | Ограничивает вечные сессии без неудобства для активных пользователей |

Выбранные параметры обеспечивают разумный баланс: реальный пользователь может ошибиться 4 раза и всё равно войти, тогда как автоматический перебор будет остановлен уже после 5 попыток.

### 3. Чем отличаются меры защиты на уровне WordPress от мер на уровне веб-сервера и ОС?

**Уровень WordPress (плагины, WAF):**
- Работают внутри самого приложения
- Могут быть обойдены при компрометации самого WordPress
- Удобны в настройке через интерфейс
- Защищают от атак на логику приложения (брутфорс, SQL-инъекции через формы, XSS)

**Уровень веб-сервера (Apache, .htaccess):**
- Работают до того как запрос достигает PHP и WordPress
- Не зависят от состояния WordPress
- Защищают от прямого обращения к файлам, листинга директорий, вредоносных запросов
- Настраиваются через конфигурационные файлы сервера

**Уровень ОС:**
- Права на файлы и папки — фундаментальный уровень защиты
- Работают независимо от любого программного обеспечения
- Защищают от несанкционированного чтения и записи файлов другими процессами
- Не могут быть обойдены через веб-интерфейс

Полноценная защита требует всех трёх уровней: компрометация одного не должна автоматически означать компрометацию остальных.

### 4. Что обязательно включать в «полный» бэкап WP и как проверить что восстановление работает?

**Полный бэкап WordPress включает:**

| Компонент | Содержимое |
|-----------|------------|
| База данных | Все записи, страницы, настройки, пользователи, комментарии |
| `wp-content/uploads/` | Загруженные медиафайлы |
| `wp-content/themes/` | Файлы тем, включая кастомные |
| `wp-content/plugins/` | Файлы плагинов |
| `wp-config.php` | Настройки подключения к БД |
| `.htaccess` | Правила сервера |

**Как проверить что восстановление работает:**

1. Создать резервную копию.
2. Намеренно удалить несколько записей и изображений.
3. Выполнить восстановление из бэкапа.
4. Проверить что удалённые данные вернулись.
5. Убедиться что сайт работает корректно после восстановления.

Именно этот процесс был выполнен в Шаге 7 данной лабораторной работы: резервная копия создана через UpdraftPlus, данные удалены, затем успешно восстановлены.
